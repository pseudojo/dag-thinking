# MCP (Model Context Protocol) Best Practices and Lessons Learned

A comprehensive, opinionated guide compiling production-grade architectural patterns, anti-patterns, and core constraints when engineering Model Context Protocol (MCP) Servers and Clients.

---

## 1. Architectural Foundation & "Tool Budgeting"

### 1.1 The 1:1 API-to-Tool Anti-Pattern
* **The Problem:** Naively exposing every endpoint of a traditional REST or GraphQL API as an individual tool leads to an overloaded "Tool Budget". High tool counts inflate token consumption, confuse the LLM's selection mechanism, and force the agent into multi-turn optimization loops just to accomplish a single high-level action.
* **The Solution:** Design tools around cohesive user intent and high-level use cases instead of backend architectures. Utilize MCP **Prompts** as macro-like capabilities to orchestrate or chain multiple underlying capabilities without exposing raw endpoints to the LLM.

### 1.2 Connection Lifecycle Management
* **The Problem:** Initializing persistent external resources (e.g., database connections, stateful API clients) on server startup prevents the MCP server from launching or listing its tools if credentials are invalid or endpoints are unreachable.
* **The Solution:** Establish resource connections dynamically *per tool invocation*. While this introduces minor latency, it ensures isolation and guarantees the server can gracefully start, be discovered, and execute smoke tests/configuration audits even in a degraded state.

## 2. Tool Definition & Interfacing Standards

### 2.1 Schema & Naming Conventions
* **Tokenization Efficiency:** LLM tokenizers (e.g., GPT-4o) process unified identifiers more efficiently. Use strict semantic structures.
  * **❌ Prohibited:** Spaces (`get Npm Package Info`), Dots (`get.Npm.Package.Info`), or Brackets (`get(Npm)Package`). These fracture tokens and cause clients to fail discovery.
  * **✅ Recommended:** `camelCase` (highly preferred for modern OpenAI models) or standard `snake_case` / `kebab-case`.
* **Parameter Leniency:** Advertise strict JSON schemas for input validation, but implement permissive, resilient parsing within the execution logic to accommodate slight structural deviations generated by the agent.

### 2.2 Semantic Tool Descriptions
* **Contextual Markup:** Do not write brief or generic descriptions (e.g., `"Executes an SQL query"`).
* **Structured Context:** Infuse operational boundaries and structural conditions using XML-like semantic tags directly inside the description field to guide the model's judgment window.
  ```javascript
  server.tool(
    "runSqlQuery",
    `<use_case>Use this tool to execute a single read-only SQL query against the production Postgres instance.</use_case>
     <important_notes>
       If a temporary database branch was created in a prior step, you MUST:
       1. Pass the branch ID to this tool unless explicitly overridden.
       2. Explicitly log to the user that execution is targeting branch [branch_id].
     </important_notes>`
  );
  ```

### 2.3 Eliminating "Not Found" Negative Steering
* **The Problem:** Returning negative strings like `Error: Module X not found` at the beginning of a tool response acts as an absolute barrier for LLM reasoning. The model frequently focuses on the explicit negative statement and discards any subsequent recovery data or alternative options provided in the same payload.
* **The Solution:** Frame responses constructively. Provide fallback information or related data without negative framing, enabling the LLM to process relevance autonomously.
  * *Contextual Exception:* This pattern must be bypassed when dealing with sensitive identity or security boundaries where fuzzy matching would trigger a data privacy violation.

## 3. Communication Channel Rules & Observability

### 3.1 Strict STDIO Boundary Controls
* **The Constraint:** For servers utilizing standard input/output (`stdio`) transport, `stdout` belongs exclusively to the JSON-RPC communication channel. Any unformatted write to `stdout` (e.g., raw `console.log`) corrupts the framing protocol and crashes the host-client interaction loop.
* **The Solution:** Route all application logs to a file-based logger (e.g., `pino` for Node.js) targeting a dedicated system log directory (`~/Library/Logs/` or `/tmp/`). Implement automated fallbacks to temporary system directories if the configured log path is unwritable.

### 3.2 Dynamic Configuration Diagnostics (The `info` Pattern)
* Expose a mandatory, lightweight diagnostic tool or subcommand (`info`) across all servers. This tool must dynamically emit:
  * The precise, non-hardcoded server version parsed directly from the environment manifest (`package.json`).
  * The validation status of required environmental variables and native dependencies.
  * Actionable structural logs rather than human-centric error messages, guiding the calling agent on how to adjust its configuration state.

## 4. Secure Deployment & Lifecycle Management

### 4.1 Immutable Containerized Packaging
* **The Challenge:** Runtime environment drift (varying Node.js/Python versions, dependency mismatches) across host environments destabilizes agent execution.
* **The Solution:** Package all production MCP servers as minimal Docker containers using explicit multi-stage builds. Ensure the container exposes standard `stdio` transport channels to natively integrate with enterprise platforms like the Docker MCP Catalog.

### 4.2 Automated Release Validation (Pre-flight Pipeline)
Implement a mandatory validation script (`prepare-release.js`) executing the following verification sequence before any server deployment:
1. **Source Control Check:** Verify active branch status, uncommitted files, and origin synchronization.
2. **Supply Chain Security:** Execute a blocking vulnerability audit (e.g., `snyk test`, `npm audit`) to reject critical or high vulnerabilities. Generate a Software Bill of Materials (SBOM).
3. **Code Quality Metrics:** Enforce strict static analysis rules. Maintain a strict size boundary: no single server file should exceed 500 lines of code (target <300 LOC) to guarantee reviewability.
4. **Smoke Testing:** Spin up an MCP Inspector instance and validate the full lifecycle: successful connection initialization, schema discovery via `List Tools`, and accurate payload execution.
